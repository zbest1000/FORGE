# Security policy

FORGE takes the security of its users seriously. This document
describes how to report vulnerabilities, our supported versions, and
the disclosure process.

## Supported versions

| Version | Status        | Security fixes |
|---------|---------------|----------------|
| `main`  | Active        | ✅ Yes         |
| `0.4.x` | Latest stable | ✅ Yes         |
| `0.3.x` | Maintenance   | ⚠️ Critical only |
| `< 0.3` | End-of-life   | ❌ No          |

The "latest stable" line is whatever is currently tagged on
`main`; we do not run a long LTS branch separately. Operators are
expected to upgrade within one minor release of a security fix.

## Reporting a vulnerability

Please report security vulnerabilities **privately** to:

- Email: `security@forge.local` *(replace with your operator's
  address before publishing this repo)*
- GitHub: open a draft security advisory at
  <https://github.com/zbest1000/FORGE/security/advisories/new>

Include:

1. A clear description of the issue and its impact.
2. Reproduction steps. A working PoC, even unfinished, is hugely
   helpful.
3. Affected versions / commits, if known.
4. The CVSS 3.1 vector if you've already scored it.

We will:

- **Acknowledge** receipt within 3 business days.
- **Assess** the report and assign a severity within 10 business
  days.
- **Fix** Critical / High issues and ship a coordinated release
  within 30 days. Medium and Low fixes follow normal release
  cadence (typically next minor).
- **Disclose** publicly via a GitHub Security Advisory + GHSA
  identifier once the fix has shipped, crediting the reporter
  unless you ask to remain anonymous.

We follow a **90-day coordinated disclosure** window. If we cannot
ship a fix within 90 days of acknowledgement, we will publish the
advisory with mitigation guidance and credit you anyway — silence is
worse than disclosure.

## Out of scope

The following are generally **not** considered vulnerabilities:

- Findings that require a malicious operator (root on the
  FORGE server) — operators have unrestricted access by design.
- Self-XSS via the demo SPA when running in `demo` mode against
  `localStorage`-only state.
- Missing rate-limiting on endpoints that are intentionally
  unauthenticated (`/api/health`, `/metrics`).
- License-tier bypass (e.g. unlocking enterprise features via
  client-side flags) — license enforcement is server-side and the
  client copy is best-effort UI.
- Reports generated by automated scanners with no PoC.

If you're not sure, send the report anyway and we'll triage it.

## Severity guidelines

We classify by **CVSS 3.1 base score**:

| CVSS  | Severity | SLA target              |
|-------|----------|-------------------------|
| 9.0+  | Critical | 7-day fix + advisory    |
| 7.0–8.9 | High   | 30-day fix + advisory   |
| 4.0–6.9 | Medium | next minor release      |
| 0.1–3.9 | Low    | next minor release      |

## Cryptographic dependencies

FORGE relies on:

- `@noble/ed25519` for license signatures.
- Node `crypto.subtle` (HMAC-SHA256, SHA-256) for the audit chain
  and approval signatures.
- `bcryptjs` (cost 10) for password storage.
- TLS termination at the reverse proxy or container edge — FORGE
  itself listens HTTP and is expected to sit behind a TLS
  terminator.

If you find a weakness in any of these dependencies' usage, that
qualifies for the standard report flow above.

## Hall of fame

Names are added to the GHSA advisory at the time of release.
