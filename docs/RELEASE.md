# FORGE — Build and Release Flow

FORGE is a build-first SPA plus Fastify server. Source-module serving
is a local development fallback only; production releases must ship
`dist/`.

The release pipeline is driven by Git tags and produces:

- Portable archives for **Linux x64**, **macOS arm64**, **macOS x64**,
  and **Windows x64** (each contains `dist/`, `server/`, pruned
  `node_modules/`, and OS-appropriate launcher scripts).
- A **multi-arch container image** (`linux/amd64`, `linux/arm64`)
  pushed to GitHub Container Registry, **signed via cosign keyless**
  (Sigstore Fulcio + Rekor), with a **build-provenance attestation**
  generated by GitHub Actions OIDC.
- A **CycloneDX SBOM** for the release.
- **SHA-256 + SHA-512 manifests** for every archive.
- A **draft GitHub Release** that aggregates everything for review.

The whole flow is in `.github/workflows/release.yml`.

## Branch flow

1. Branch from `main`.
2. Keep commits scoped to one logical change.
3. Before opening or updating a PR, run:
   ```bash
   npm run release:check
   ```
4. Push and open a PR.

## Release verification (every PR)

`npm run release:check` runs:

1. `npm run build` — creates the hashed Vite bundle in `dist/`.
2. `npm test` — runs the Node test suite.
3. `npm run smoke:built` — starts Fastify against the built bundle on
   port 3100, checks `/api/health`, and verifies `/` serves the
   bundled asset paths.

CI (`.github/workflows/ci.yml`) does the same on every push and PR
across **Linux + macOS + Windows** runners and **Node 20 LTS + 22**.

## Cutting a release

A release is just a tag push. The `release.yml` workflow does the rest.

```bash
# 1. Bump the version in package.json (kept in sync with the tag)
npm version patch    # or minor / major
# This creates a "vX.Y.Z" annotated tag.

# 2. Push the commit and the tag.
git push origin main
git push origin --tags
```

The pushed tag triggers four parallel jobs:

| Job          | What it does                                                           |
| ------------ | ---------------------------------------------------------------------- |
| `archives`   | Cross-platform matrix (linux/macos arm64+x64/windows). Builds, prunes dev deps, stages launchers, packs `.tar.gz` / `.zip`, computes checksums. |
| `sbom`       | Runs `npm run sbom` and uploads `forge-VERSION.cdx.json`.              |
| `container`  | Multi-arch image build → push GHCR → cosign sign → attest provenance.  |
| `release`    | Aggregates artifacts, attests archive provenance, opens draft release. |

The release is created as a **draft** so a human can review the
artifacts and the auto-generated changelog before flipping it to
public.

### Verifying a release

Anyone — customer, auditor, mirror operator — can verify a release end
to end without trusting the GitHub UI:

```bash
# 1. Archive integrity
sha256sum -c SHA256SUMS

# 2. Archive provenance (Sigstore)
gh attestation verify forge-1.0.0-linux-x64.tar.gz \
  --repo zbest1000/FORGE

# 3. Container image signature
cosign verify ghcr.io/zbest1000/forge:1.0.0 \
  --certificate-identity-regexp 'https://github\.com/zbest1000/FORGE/' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com

# 4. Container image provenance attestation
gh attestation verify oci://ghcr.io/zbest1000/forge:1.0.0 \
  --repo zbest1000/FORGE
```

## Local archive packing (dry-run)

Engineers and field teams can produce a target-specific archive locally
without waiting for CI:

```bash
npm run build
npm run release:archive -- --target linux-x64
# → build/forge-VERSION-linux-x64.tar.gz
# → build/forge-VERSION-linux-x64.tar.gz.sha256
# → build/forge-VERSION-linux-x64.tar.gz.sha512
```

`scripts/release/pack-archive.js` mirrors the staging layout the CI job
uses, so a locally produced archive and a CI-produced archive should
have the same files, just without a vendor signature.

## Production rules

- `npm start` requires `dist/index.html`. If absent and
  `FORGE_SERVE_SOURCE` is not set, the server exits with a helpful
  error rather than silently serving source files in production.
- Strict secrets validation in `server/config.js`: `FORGE_JWT_SECRET`
  and `FORGE_TENANT_KEY` must be ≥ 32 characters in production /
  strict mode, and `FORGE_CORS_ORIGIN=*` is rejected.
- The Vite production build:
  - lazy-loads heavy screens (`src/screens/drawingViewer.js` and
    friends) so the initial bundle stays small (≈ 65 kB gzipped).
  - emits per-vendor manual chunks (`vendor-web-ifc`, `vendor-three`,
    `vendor-pdfjs`, etc.) so caches survive deployments.
  - strips the dev import map from `index.html` so the production CSP
    can drop `https://esm.sh` from `script-src` / `connect-src`.

## Docker

The `Dockerfile` is multi-stage:

1. **builder** (`node:20-bookworm-slim` + build tools): runs
   `npm ci`, `npm run build`, then `npm prune --omit=dev`.
2. **runtime** (`node:20-bookworm-slim` + `tini`): copies the pruned
   `node_modules`, the built `dist/`, `server/`, and license public
   key. Runs as the `node` user with a HEALTHCHECK against
   `/api/health`.

`docker-compose.yml` includes a `distributed` profile that spins up
Postgres, NATS, an OTel collector, a worker replica, and Mosquitto for
the HA deployment story (`ops.ha` license feature).
